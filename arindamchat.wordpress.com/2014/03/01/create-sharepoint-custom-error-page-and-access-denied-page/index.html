Create SharePoint Custom Error Page and Access Denied Page

I want to show here how easily you can set a custom error page and custom access denied page in your SharePoint site.
Approach 1
 
In the first approach I’m showing how you can just use the OOB error page and OOB access denied page in your application with small change.
Here I assume that you have a SharePoint Project to deploy your custom pages to 14 hive. Follow below steps
·         Create a new SharePoint project
·         Add a “SharePoint Mapped” folder for Layouts (see below)

 

·         Then copy the OOB error page and access denied page and add to your solution from C:\Program Files\Common Files\Microsoft Shared\Web Server Extensions\14\TEMPLATE\LAYOUTS
·         Rename then as CustomError.aspx and CustomAccessDenied.aspx.

 

You can do very small amount of changes here like below
For custom error page you make some changes here, be careful not to remove any control:

Advantage:
·         You don’t need any coding, the page is ready to use.
·         The page will have the same look and feel as other pages on your site as per master page set. So you can stop the nasty white error page with co-relation ID.
·         Most importantly you can still show the co-relation ID in the error page, this will be immensely helpful for support guys.
Disadvantage:
Since no major coding is possible (you can do some coding using script tag) since there is no code behind you can’t do much customization
 
Approach 2
 
Here I’ll show how you can create a more specific custom error page and show.
In the way you create the SharePoint project above, add an “Application page” to the Layouts folder of you project.
If not already added, add below in the HTML page
<asp:Content ID=”PageHead” ContentPlaceHolderID=”PlaceHolderAdditionalPageHead” runat=”server”>
<style type=”text/css”>

#s4-leftpanel-content {display:none;}
</style>

Disadvantage
Co-relation ID can’t be shown in the page when any error occurs.
Deployment and activation
 
Since the pages are added to Layouts folder they will be deployed when you deploy the WSP. But you need to override the OOB custom error page and access denied page setting.
PowerShell:
 
You can use PowerShell, this is the easy way of doing and suitable if you are deploying your code through PowerShell in Production environments.
Use below PowerShell
Set-SPCustomLayoutsPage -Identity Error -RelativePath /_layouts/YourWeb/CustomError.aspx -WebApplication  $portalUrl
Set-SPCustomLayoutsPage -Identity AccessDenied -RelativePath /_layouts/YourWeb/CustomAccessDenied.aspx -WebApplication  $portalUrl
Feature:
 
You can also use feature to set this as below
public override void FeatureActivated(SPFeatureReceiverProperties properties)
{
    stringCustomErrorPage = “/_layouts/CustomError.aspx”;
    SPWebApplication webApp = (SPWebApplication)properties.Feature.Parent;
    if(!webApp.UpdateMappedPage(SPWebApplication.SPCustomPage.Error, CustomErrorPage))
    {
        throw newSystem.ApplicationException(“Cannot replace the current error page.”);
    }
    webApp.Update(true);
}
public override void FeatureDeactivating(SPFeatureReceiverProperties properties)
{
    if(!webApp.UpdateMappedPage(SPWebApplication.SPCustomPage.Error, null))
    {
        throw newSystem.ApplicationException(“Cannot replace the current error page.”);
    }
    webApp.Update(true);
}
