Agile Methology

Introduction

During the 90s a number of different people realized that things had somehow changed. These people became interested in developing software methodologies that were a better fit with the new business environment – more nimble, easier to maneuver through a turbulent business environment. Although the details of these methodologies differ, they all share certain underlying principles, to the extent that these methodologies are often now grouped under the title “agile methodologies”.

Taichi Ohno was the first to implement this technique in Toyota production system.

The Fundamental agile principle is: Eliminating waste.

Problem with older approaches

There have been a lot of Software Development Processes created over the years. With exception of Code-and-Fix, these processes have few things in common – they assume that software development is analogous to defined industrial process; they are based on typical engineering processes, Physical engineering processes consist of distinct phases like design phase, planning for the construction phase etc. and same were being followed by software development process also.

But the problem in software development is that, not all requirements can be defined up front as is needed in BDUF (Big Design Up Front) approach. Also, a tendency to overbuild (add features that are never used) and over-engineer the infrastructure exists, and the most common complaint is “the problem with this project is that the user keeps changing his mind”.

In traditional methodologies it is very expensive to make changes after a certain point.
While predictability may be very desirable, it can only be achieved by seriously compromising other qualities – particularly the fit of final results to the real (emerging) requirements.

Why we need Agile

Given these flaws in traditional software development methodologies, it’s worthwhile asking why they ever worked, and if they worked in the past, why abandon them now. Has something changed?
Fundamentally, the business world has changed. Existing businesses change direction just as quickly in response. Product cycles are shorter, and consumers are fickle. It’s no surprise that methodologies that were adequate in the 70s, when product and business cycles were much longer, are being stretched to the point of failure in the 80s and 90s.

So why use Agile? Here are the points

• A growing number of software projects operates in extremely dynamic environments.
• Project team challenges often include the integration of leading edge technologies.
• The need to respond to erratic requirements changes.
• Increasingly short time to market cycles.
• Projects may have relatively clear objectives, but the specific requirements can be volatile and evolving as customers
and development teams alike explore the unknown.
• The projects with the above attributes are called high-exploration factor projects, often fall short of expectations when developed using overly rigorous, plan-driven (Waterfall) methods. To implement these kinds of projects with the above attributes, we use Agile.

Agile approach is recommended when:

Domain & Technology are evolving
Requirements change
Short timelines for delivery

Spirit of Agile

• The highest priority to satisfy the customer through early and continuous delivery of Software

• Welcome changing requirements. Agile processes harness changes for customer’s competitive advantages

• Deliver-working software frequently, from couple of months to couple of weeks, with a preference for shorter time scale

• Business people and developers must work together throughout the project

• Build projects around motivated people. Give them the environment and support to get the job done

• Most effective method of getting information around in the team is “face-to-face” conversations

• Working software is the primary measure of progress

• Maintain a constant work pace

• Continuous attention to technical excellence

Existing Agile Methodologies & selection of Agile Methods

Agile Methods have much in common, such as what they value, but they also differ in the practices they suggest.

• Dynamic System Development Method (Dane Faulkner)
• Adaptive Software Development (Jim Highsmith)
• Crystal (Alistair Cockburn)
• SCRUM (Ken Schwaber)
• XP (Kent Beck)
• Lean Software Development (Mary Poppendieck)
• Feature Driven Development (Jeff DeLuca)

Key concepts

Story cards

At the start of each iteration, customers, managers, and developers meet to flesh out, estimate, and prioritize requirements for the next release. The requirements are called “user stories” and are captured on “story cards” in a language understandable by all parties.

Refactoring

As developers work, the design should be evolved to keep it as simple as possible.

Pair Programming

Two developers sitting at the same machine write all code, some finding of pair programming is

· Economics: 15% higher, not 100%
· Improved design quality
· Continuous reviews
· Collective code ownerships
· Learning and problem solving
· Improves team communication
· More people understand each piece of the system
· Nominated several times as an improved way of developing software

Continuous Integration

Developers integrate new code into the system as often as possible. All functional tests must still pass after integration or the new code is discarded. The salient points are

· Keep a single place where all the source code lives and from where anyone can obtain the current sources from (and previous versions)
· Automate the build process so that anyone can use a single command to build the system from the sources
· Automate the testing so that anyone can run a suite of test systems with a single command
· Make sure anyone can get a current executable which is best executable so far

Collective ownership

The code is owned by all developers, and they may make changes anywhere in the code at anytime they feel necessary.

Stand up meetings

These meetings should be held on daily basis, they are short, 15 to 30 minutes long; should address obstacles but not define solutions

Purpose:
· Communicate daily status, progress, plans to team and any observers
· Identify obstacles and remove them
· Set focus for the rest of the day

Agenda: Same 3 questions asked –
· What is incomplete or backlog?
· What have you accomplished between the last meeting and now?
· What will you be doing between now and the next meeting?

Benefits: Improves communication and helps integration of the team

Retrospectives: Looking Back to move forward,

A project retrospective is an honest and systematic look back at a recently completed project.
The objective is to identify lessons in a project and identify actions that can be taken to make the next project more successful, a retrospective must be safe for the attendees to voice their opinions. This is often difficult and takes time to get the team to open up.

A project Execution Model

Different Phases in Agile Development are Strategy, Planning, Execution and Release

Strategy
To verify that the project is worth doing and the entire team independently understands the project direction.

Activities in Strategy Phase:
Idea Generation :The business identifies product opportunities and drives formulating the business and financial plans by collaborating with support teams from other business areas and technology. Involvement of the appropriate individuals from horizontal teams and technology.

Rationalization : In this phase the product owner presents new project or product idea to the innovation council.

Prioritization :Upon approval the product owner works with the development team to ensure proper priorization.

Pre-Project Planning : This phase begins to determine the sequencing / project trade-offs and project requirements for resources, and to provide a project vision document. A product manager prepares the vision document. The technical manager / director will assist in sequencing the project and the PM begins the work on the project.

Planning

Planning phase sets the priority for the use cases or stories and an agreement of the contents of the releases will be made. Each sprint (iteration) will work either with a subset of use cases or with a vertical slice across all the use cases. The requirements will be rated on the scale of the priority and business criticality. The pieces of the requirements, which are rated high on the scale, will be delivered in the early sprints. Release Planning
Release planning is the set of scoping and scheduling activities conducted by client and the development team, at the beginning of each release cycle

Sprint (Iteration) Planning
Sprint is a 2-4 week time-boxed period that:
· Produces a small, tested, integrated subset of the solution
· Produces potentially deployable functionality and functionality is validated by clients
and used as feedback for the next iteration

This time-boxed planning involves low-level plan of the tasks and the functionalities to be implemented i.e. elaborated, designed, coded and tested.

Activities in Strategy Phase:

Project Kick off : The entire team discusses about the project. Introduce the business case to the team.

User Stories : Requirements of the project broken into sizeable pieces. Each are independent, hold value, can be estimated and tested.

Backlog :An ordered collection of deliverables for a project. The requirements are written in User-story format.

Project Release Plan :Used to determine the time frames for implementing various features. The implementation can be staged if it makes sense.

Execution

Activities in Execution Phase:

Iteration Plan :Select User stories, Leveling User Stories, Break into Tasks & Estimate Tasks.

Iteration Execution :Monitor & control iterations, Analysis & Design, Code, Test, Review, Integrate & Regress, Certify & Accept.

Iteration Closeout :Demonstrate, Measure Team Velocity, Retrospective (Lessons Learned), Inspect, and Adapt & Plan.

Release

The activities in the Release Phase are:

Merge to Release:Merge the project code into the main Release branch.

Functional Test :The functional testing is used to help insure the integrity, functionality and “fitness” of development team software products before the software products are placed into Release Regression testing.

Regress/Accept Test:The release regression team executes a planned set of featured acceptance tests for the entire release. The release co-coordinator provides the regression test plan for manual and automation QA testing teams.

Implementation :The implementation activities are the final phases of the Release Cycle and place the project product into production.

Post Imp :Verification of the project product in the production environment.
